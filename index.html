什么是单元测试

首先，让我们来了解一下单元测试是什么？以及为什么我们需要编写单元测试。

以下是维基百科关于单元测试的定义：

单元测试是一种软件测试方法，用于测试独立的源代码单元，来确定这些代码是否满足需求。

这种思想是为每个重要的功能或方法写一段测试代码。它使得你能相对快速地检查出最新的代码改动是否导致了回归（regression）。例如，新的错误出现在已经测试过的程序中，并且会使得发现和解决这些错误变得更加容易。

单元测试的作用

单元测试有必要吗？

1.对是否有必要写单元测试的疑惑

  • 没有价值：不做单元测试照样开发，没有什么问题；
  • 浪费时间：写单元测试需要大量的时间，多写了很多代码，还不如写具体的实现能看到明显的效果，单元测试可能耽误开发进度；
  • 无法测试：比如无返回值的方法、UI等；

2.不写单元测试会存在的问题

  • 要有足够的耐心：改一个参数，需要重新运行一遍程序；
  • 要有足够多的时间：必须要等到测试发现Bug后才改善；
  • 修改Bug会造成Bug：觉得不会对程序造成影响，结果还是造成了Bug，QA测试不到，到上线才发现Bug；
  • Bug太多，程序很难稳定：上报的大多数异常问题，都是因为没有做好容错导致的，比如空指针、被除数为0、数组越界等；

3.单元测试能够解决的问题

  • 效率：如果没有单元测试就必须把程序运行起来，一次几分钟；运行一次单元测试，最多几分钟，cover得比较全面，相比执行程序，效率要高很多。
  • 质量：对于每个最小单元，针对不同输入对应的输出有与预期做对比，能够减少因为参数导致的异常问题，同时提测和发布版本的时候，有信心；
  • 提升设计能力：为了每个单元都可测，需要将每个方法拆得尽量独立，如果不拆得足够独立，就无法测试，间接可以提高程序设计能力；
  • 代码重用以及重构：跑过单元测试的代码，稳定性能够得到保证，可以在其它项目或者项目重构时重复利用；在重构时有单元测试可以有信心；
  • 缩短测试周期：程序自测（开发人员写单元测试、手动跑基本功能、跑monkey都属于自测）可以提高产品提测的质量，避免返工；同时核心功能的稳定有助于缩短黑盒测试的周期。

Android中的单元测试

Android有2种单元测试方法：

  1. Local unit tests: 这是只用JVM来执行的测试，主要是用来测试不与操作系统交互的业务逻辑。
    • 特点：是执行快，几秒钟的时间。不依赖设备或虚拟机，能够直接在PC上执行。
    • 代表框架：JUnit
  1. Instrumented unit tests: 用于测试与Android API互联的逻辑。他们是在物理设备或者模拟器上执行。
    • 特点：执行慢，需要打包运行在手机或模拟器，几分钟的时间。
    • 代表框架：Espresso

单元测试的步骤

一个单元测试大概可以分为三个部分：

  1. setup: 即new 出待测试的类，设置一些前提条件
  2. 执行动作: 即调用被测类的被测方法，并获取返回结果
  3. 验证结果: 验证获取的结果跟预期的结果是一样的

一个类的方法分两种，一种是有返回值的方法，一种是没有返回值的方法，即void方法。对有返回值的方法测起来很容易，对于没有返回值的方法呢？

比如说有一个DataActivity，它有一个public void loadData()方法，会去调用底层的DataModel类，异步执行一些网络请求，然后更新用户界面。
最直接的反应是，调用loadData()方法，然后过一段时间，验证界面得到了更新。然而这种方法叫集成测试，而不是单元测试。因为它涉及到DataModel、网络服务器，以及网络返回正确时Presenter的内部处理，等等。集成测试有它的必要性，但不是我们应该关注的地方，也不是最有价值的地方。我们应该关注的是单元测试，关于这一点，有一个Test Pyramid的理论。


Test Pyramid理论基本大意是，单元测试是基础，是我们应该花绝大多数时间去写的部分，而集成测试等应该是冰山上面能看见的那一小部分。
那么对于这个case，正确的单元测试方法，应该是去验证loadData()方法调用了DataModel的某个请求数据的方法，同时传递的参数是正确的。“调用了DataModel的方法，同时参数是。。。” 这个才是loadData()这个方法的“返回结果”。

Mock的概念

要验证某个对象的某个方法得到调用了，就涉及到mock的使用。

mock就是创建一个虚假的、模拟的对象。如果要测试的目标对象依赖关系比较多，需要解除依赖关系，以免测试用例过于复杂。

在测试环境下，用来替换掉真实的对象。这样可以达到两个目的：

  1. 可以随时指定mock对象的某个方法返回什么值，或执行什么样的动作。
  2. 可以验证mock对象的某个方法有没有得到调用，或者是调用了多少次，参数是什么等。

在测试环境中使用mock： 依赖注入

接下来的问题是，如何在测试环境下，把Presenter中的DataModel换成mock的对象，而在正式的代码中，DataModel又是正常的对象呢？
两种方案:

  1. testing product flavor: 把需要mock的类写一份mock的implementation，然后通过factory提供给client。这个factory的接口在testing flavor和正式的flavor里面是一样的。
  2. 依赖注入: Dagger2。关于依赖注入，请看下面的图。在测试环境，创建一个生产mock DataModule的TestModule，使用TestModule来创建Component。

那么我们选择哪种框架来做单元测试呢？

单元测试的要点:

  • 执行快，效率高；
  • 不依赖Android设备，可以直接由Jenkins周期性执行；

但是Android中，编写代码需要面对的是组件、控件、生命周期、异步任务、消息传递等，虽然本质是SDK主动执行了一些实例的函数，但创建一个Activity并不能让它执行到resume的状态，因此需要JUnit之外的框架支持。

我们现在用到了哪些框架进行测试？

  • JUnit:
    用一定的架构，比如MVP等等，将安卓相关的代码隔离开了，中间的Presenter或Model是纯java实现的，可以在JVM上面测试。

  • Mockito:
    Tasty mocking framework for unit tests in Java

  • Robolectric:

    Robolectric这个框架基本可以理解为在JVM上面实现了一套安卓的模拟环境，同时给安卓相关的类增加了其他一些增强的功能，以方便做单元测试，使用这个框架，我们就可以在JVM上面跑单元测试的时候，就可以使用安卓相关的类了。

ContentProvider的使用

ContentProvider是四大组件中使用最少的一个，当我们需要为其他应用程序提供数据时，我们就需要使用ContentProvider了。首先理解这其中ContentProvider是一个内容提供的服务端，其他使用ContentProvider提供的数据的应用为客户端。
ContentProvider中的内容是像一个关系型数据库一样的结构。每一行代表数据库表中的一行数据，每一列代表某一主键的数据。
ContentResolver方法可提供持续存储的基本CRUD功能。例如从用户字典提供程序中获取字词，需要调用ContentResolver.query()。

mCursor = getContentResolver().query()(
    UserDictionary.Words.CONTENT_URI,       //Uri映射至ContentProvider中的表
    mProjection,                            //查询的字段
    mSelectionClause,                       //查询条件
    mSelectionArgs,                         //查询参数
    mSortOrder                              //排序条件
);

基本操作和数据库的操作很像，其中Uri是这样content://authority/table：
content:// 架构
authority ContentProvider的授权
table 表的路径
许多ContentProvider都允许将ID值追加到URI末尾来访问表中的单个行。例如，要从用户字典中检索_ID为4的行，则可以使用:

Uri singleUri = ContentUris.withAppendedId(UserDictionary.Words.CONTENT_URI, 4);

创建TestProvider继承ContentProvider，实现query、insert、delete、update方法:

@Override
public Cursor query(Uri uri, String[] projection, String selection, String[] selectionArgs, String sortOrder) {
    final SQLiteDatabase db = mOpenHelper.getReadableDatabase();

    Cursor cursor = null;
    switch (buildUriMatcher().match(uri)) {
        case TEST:
            cursor = db.query(TestContract.TestEntry.TABLE_NAME, projection, selection, selectionArgs, sortOrder, null, null);
            break;
    }
    return cursor;
}

@Override
public Uri insert(Uri uri, ContentValues values) {
    final SQLiteDatabase db = mOpenHelper.getWritableDatabase();
    Uri returnUri;
    long _id;
    switch (buildUriMatcher().match(uri)) {
        case TEST:
            _id = db.insert(TestContract.TestEntry.TABLE_NAME, null, values);
            if (_id > 0) {
                returnUri = TestContract.TestEntry.buildUri(_id);
            } else {
                throw new android.database.SQLException("Failed to insert row into" + uri);
            }
            break;
        default:
            throw new android.database.SQLException("Unknown uri: " + uri);
    }
    return returnUri;
}

private final static int TEST = 100;

static UriMatcher buildUriMatcher() {
    final UriMatcher matcher = new UriMatcher(UriMatcher.NO_MATCH);
    final String authority = TestContract.CONTENT_AUTHORITY;

    matcher.addURI(authority, TestContract.PATH_TEST, TEST);
    return matcher;
}

不要忘了在Manifest中声明此ContentProvider:

<provider
        android:authorities="com.warpath.contentprovidertest"
        android:name=".contentProvider.TestProvider"
        android:readPermission="me.warpath.READ"
        android:exported="true"
        />

authorities就是上面说的授权，readPermission代表客户端读数据时需要申请的权限，exported使provider可以被其他应用使用。至此，这个ContentProvider就可以在应用内被使用了。

ContentValues contentValues = new ContentValues();
contentValues.put(TestContract.TestEntry.COLUMN_NAME, "warpath");
contentValues.put(TestContract.TestEntry._ID, System.currentTimeMillis());
getContentResolver().insert(TestContract.TestEntry.CONTENT_URI, contentValues);

Cursor cursor = getContentResolver().query(TestContract.TestEntry.CONTENT_URI, null, null, null, null);
try {
    Log.e("ContentProviderTest", "total data number = " + cursor.getCount());
    cursor.moveToFirst();
    Log.e("ContentProviderTest", "total data number = " + cursor.getString(1));
} finally {
    cursor.close();
}

如果要在其他应用中使用，需要在Manifest中声明一个权限:

<permission android:name="me.warpath.READ"
android:protectionLevel="normal"/>

其中protectionLevel声明此权限为低风险权限，在安装时不需要用户确认。
这时在客户端中我们就可以使用ContentProvier提供的数据了。
在Manifest中申请权限:

<uses-permission android:name="me.warpath.READ"/>

使用时：

private static final Uri CONTENT_URI = Uri.parse("content://com.warpath.contentprovidertest/test");

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);

    Cursor cursor = getContentResolver().query(CONTENT_URI, null, null, null, null);
    cursor.moveToFirst();

    TextView tv = (TextView) findViewById(R.id.tv_content);
    tv.setText(cursor.getString(1));

    cursor.close();
}

ContentResolver封装了系统提供的ContentProvider比如联系人、日历和第三方应用提供的ContentProvier。

Fragment的生命周期详解

发表于 2017-07-12

Fragment的生命周期详解

先看官方图：

public void onAttach(Context context)

当fragment一旦被关联到Activity时被调用，是在被add或replace后。onCreate(Bundle)会接着被调用，也就是说fragment的构造函数调用时，还没有发生onAttach。

另外，如果你复写了这个方法，必须调用super.onAttach(context)。从该方法开始就可以通过Fragment.getActivity方法获取与Fragment关联的窗口对象，但因为Fragment的控件未初始化，所以不能够操作控件。

public void onCreate(Bundle savedInstanceState)

在调用完onAttach之后，onCreateView之前调用。但是在Activity被重新创建，同时Fragment设置了RetainInstance时，onCreate不会被调用。注意这个方法被调用的时候Activity也有可能在创建中，所以不能在此时信赖Activity的View结构树创建完成。如果你想在Activity创建完毕后做一些操作，请使用onActivityCreated(Bundle)。
同样复写这个方法的时候要调用父方法。

public View onCreateView(LayoutInflater inflater, ViewGroup container, Bundle savedInstanceState)

被调用使得Fragment可以初始化界面，当然也可以返回null。在onCreate(Bundle)和onActivityCreated(Bundle)中间调用。
如果你返回了一个View,当Veiw被销毁的时候onDestroyView()会被调用。
可以添加没有UI的Fragment，请使用add(Fragment, String)从Activity添加Fragment(为Fragment提供一个唯一的字符串“标记”)。这会添加片段，但由于它并不与Activity布局中的视图关联，因此不会收到对onCreateView()的调用。例如可以添加一个没有UI的DialogFragment作为登录验证的程序，在DialogFragment中做登录校验，在任何界面可以使用。

public void onViewCreated(View view,Bundle savedInstanceState)

在onCreateView返回之后马上被调用，但是在saved state被恢复到View之前。给了子类一个在View hierarchy创建之后初始化自己的一个机会。此时，Fragment的View已经和Activity中的View关联在一起。

public void onActivityCreated(Bundle savedInstanceState)

在Activity的onCreate方法返回之后，系统会立马调用该方法，表示窗口已经初始化完成。这里可以做最后的初始化操作，比如恢复saved state。这个时候也可以调用Fragment的setRetainInstance(boolean)去保持实例。
从这个时候开始，就可以在Fragment中使用getActivity().findViewById(id)来操控Activity中的View了。

public void onStart()

当系统调用该方法的时候，fragment已经显示在UI上，但还不能进行互动，因为onResume方法还没执行完。

public void onResume()

该方法为Fragment从创建到显示Android系统调用的最后一个生命周期方法，调用完该方法，Fragment就可以与用户互动了。

public void onPause()

同Activity

public void onStop()

当onStop返回的时候，fragment将从屏幕上消失。

public void onDestoryView()

该方法的调用意味着在onCreateView中创建的视图都将被移除。允许Fragment清理和View相关的资源。

public void onDestroy()

Android在Fragment不再使用时会调用该方法。

public void onDetach()

为Fragment生命周期中的最后一个方法，当该方法执行完后，Fragment与Activity不再有关联。

View的基础知识

发表于 2017-07-12

View的参数

mLeft View的左上角离父控件左边的距离
mTop View的左上角离父控件上边的距离
mRight View的右下角离父控件左边的距离
mBottom View的右下角离父控件上边的距离

控件的width = mRight - mLeft，height = mBottom - mTop。
width和height都可以大于屏幕。

x = mLeft + translationX
y = mTop + translationY

x和y就是getX()和getY()方法的返回值，都可以为负数或者大于父控件。
translationX为正时向右移动，translationY为正时向下移动。

View的滑动

View#scrollTo/scrollBy

可以满足滑动需求，但是内容无法滑出View的边界
mScrollX View的内容在X轴上的偏移，注意是View的内容而不是View。
mScrollY是View的内容在Y轴上的偏移。

public void scrollTo(int x, int y)
设置View在X、Y轴上的偏移量，这个滑动是瞬间完成的。

public void scrollBy(int x, int y)
查看源码便知scrollBy是调用了scrollTo使View的内容有相对偏移。

当x为正时向左边滑动，为负时向右边滑动。当y为正时向上滑动，为负时向下滑动。和XY坐标轴的方向刚好相反。
使用scrollTo和scrollBy的滑动都是瞬间完成。如果想做出弹性滑动的效果，需要借助一个类Scroller。 

Scroller mScroller = new Scroller(context);

public void smoothScrollTo(int destX, int destY) {  
    int scrollX = getScrollX();
    int scrollY = getScrollY();
    int deltaX = destX - scrollX;
    int deltaY = destY - scrollY;
    mScroller.startScroll(scrollX, scrollY, deltaX, deltaY, 3000);
    invalidate();
}

@Override
public void computeScroll() {
    if (mScroller.computeScrollOffset()) {
        scrollTo(mScroller.getCurrX(), mScroller.getCurrY());
        postInvalidate();
    }
}

Scroller并不会使View产生滑动，它只是当你告诉它要在一定的时间内将View滚动一定距离，任何时刻View存在的位置。具体的滑动，需要View调用自己的scrollTo方法实现。

Animation#TranslationX/TranslationY

可以任意滑动，但是在3.0以下会影响点击事件
弹性滑动使用ValueAnimator#AnimatorUpdateListener来完成。使用属性动画更方便。 

ObjectAnimator.ofFloat(mScrollView, "translationX", -100).setDuration(3000).start();

translationX为负时向左滑动，translationX为正时向右滑动。translationY为负时向上滑动，translationY为正时向下滑动。

real Layout changes

并不常用，改变LayoutParams。使用较为复杂。

关于Activity的生命周期

发表于 2017-07-12

关于Activity的生命周期

从Activity的创建说起，onCreate一般需要我们处理，来设置UI和进行初始化。但是onCreate里面尽量少做事情，避免程序启动太久都看不到界面。

于此对应，Activity的最后一个回调函数是onDestroy()。如果Activity含有在onCreate调用时创建的后台线程，或者是其他有可能造成内存泄漏的资源，则应该在onDestroy()时进行资源清理，杀死后台线程。

当onCreate只是完成一个中转，在onCreate里面调用了finish方法时onPause和onStop方法并不会被调用。而是直接调用onDestroy()方法。

官方Training上说，通常应该在onPause()方法里做以下事情:

  • 停止动画或者是其他正在运行的操作，比如视频播放
  • 提交在用户离开时期待保存的内容（例如邮件草稿）
  • 释放系统资源，如broadcast receivers,sensor(比如GPS)，或者是其他任何会影响到电量的资源

这里有一个疑问是Guide里面写：

您可以在onStart()中注册一个BroadcastReceiver以监控影响UI的变化，并在用户无法再看到您显示的内容时在onStop()中将其取消注册。

到底在哪里注册BroadcastReceiver？应该是onStart()和onStop()，因为onDestroy()是并不保证被执行的。所以我们在onPause()时应该快速处理。例如资源的释放，动画的暂停，视频的暂停等等。不应该使用onPause()来保存用户改变的数据到永久存储。

我们应该避免在onPause()时执行CPU-intensive的工作，例如写数据到DB，因为它会导致切换到下一个Activity变得缓慢（应该把那些heavy-load的工作放到onStop()去做）

还有我们经常使用的EventBus应该在哪里注册与反注册。EventBus的官方DOC里面是在onStart()和onStop()中进行。但是有时候如果Activity自己调用finish结束，就会造成内存泄漏。

再来说说什么情况下onDestory并不会被执行。当Activity调用onStop()方法，activity不再可见。一旦Activity停止，系统会在需要内存空间时摧毁它的实例。极端情况下，系统会直接杀死我们的app进程，并不执行activity的onDestroy()回调方法，因此我们需要在onStop()释放资源，从而避免内存泄漏。

如果我们在onStop里面做了清除的操作，就该在onStart里面重新把那些清除掉的资源重新创建起来。

跳转到其他的activity或者是点击Home都会导致当前的activity执行onSaveInstanceState，因为这种情况下的activity都是有可能会被destory并且是需要保存状态以便后续恢复使用的。而从跳转的activity点击back回到前一个activity，那么跳转前的activity是执行退栈的操作，所以这种情况下是不会执行onSaveInstanceState的，因为这个activity不可能存在需要重建的操作。

onRestoreInstanceState()方法会在 onStart() 方法之后执行. 系统仅仅会在存在需要恢复的状态信息时才会调用 onRestoreInstanceState() ，因此不需要检查 Bundle 是否为null。

所以onSaveInstanceState和onRestoreInstanceState并不是成对出现的。

View的基本知识

1、位置属性

View有四个位置参数，分别是left、top、right、bottom。这四个参数相对于父控件。
通过getLeft() 、getTop()、getRight()、 getBottom()方法获取。
分别代表View左上角离左边、上边的距离和View右下角离上边、左边的距离。

2、点击事件
MotionEvent就是点击屏幕之后Android系统需要处理的对象。
典型的时间类型有:
ACTION_DOWN: 手指刚接触屏幕
ACTION_MOVE: 手机在屏幕上移动
ACTION_UP: 手指从屏幕上松开的一瞬间

TouchSlop是系统所能识别出的被认为是滑动的最小距离。

3、View的滑动

  1. 使用scrollTo/scrollBy滑动，scrollTo的参数为绝对位置，scrollBy的参数为相对位置
  2. 使用动画，操纵View的translationX或者translationY。既可以使用传统的View动画，也可以使用属性动画
  3. 改变布局参数

滑动方式的对比:
scrollTo: 适合对View内容的滑动
动画: 适用于没有交互和实现复杂的动画效果
改变布局参数: 操纵复杂，适用于有交互的View

Activity的生命周期比较熟悉，但是Fragment的生命周期是什么样呢？
和Activity一样，Fragment也存在如下状态:

  • 活动状态:Fragment位于前台，用户可见，可以获得焦点
  • 暂停状态:Fragment位于前台，该Fragment依然可见，不能获得焦点
  • 停止状态:该Fragment不可见，失去焦点
  • 销毁状态:该Fragment被完全删除，或该Fragment所在Activity被结束
